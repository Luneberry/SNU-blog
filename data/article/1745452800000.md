<h1>통합 데이터셋 구축 — load_split() 단일 진입점</h1>

<p style="color:#6b7280; font-size:0.95em;">실험이 늘어날수록 데이터 경로 하드코딩이 쌓였습니다. 어느 실험이 어떤 데이터를 썼는지 추적하기 어려워지면서, 재현성 문제가 생겼습니다. 이를 해결하기 위해 <code>data/loader.py</code>를 중심으로 데이터 접근을 표준화했습니다.</p>

<hr/>

<h2>1. 문제: 흩어진 경로, 불투명한 분할</h2>

<p>프로젝트 초기부터 쌓인 데이터가 여러 곳에 흩어져 있었습니다.</p>

<ul>
<li><code>Project 0/data/</code> — 원본 JSON/CSV</li>
<li><code>Project 1/Train/cache/</code> — 28개 PKL (학습/테스트 분할 포함)</li>
<li><code>Project 1/experiments/*/cache/</code> — 실험별 캐시</li>
</ul>

<p>각 실험 코드가 자기 경로를 하드코딩해서 썼기 때문에, "이 실험에서 v2 테스트셋을 썼나 v1을 썼나"를 코드를 직접 열어야 알 수 있었습니다.</p>

<h2>2. 해결: data/ 통합 + load_split() API</h2>

<p><code>dog/data/</code> 아래 정본 데이터를 모으고, 기존 경로는 역방향 심볼릭 링크로 유지해 기존 코드를 수정 없이 동작하게 했습니다.</p>

<pre style="background:#f3f4f6; padding:12px; border-radius:8px; font-size:0.9em; overflow-x:auto;"><code>from data.loader import load_split

# 단일파행 3-class (v2 테스트셋)
train, test = load_split('category2_single', version='v2')

# 신경계 vs 정형외과 (binary)
train, test = load_split('category1_neu', fold=0, label_scheme='binary')

# 메타 정보까지 포함한 rich dict
train, test = load_split('category2_single', version='v2', rich=True)</code></pre>

<h2>3. 발견한 것들</h2>

<p>데이터를 정리하면서 몇 가지를 발견했습니다.</p>

<ul>
<li><strong>valid_label.json 경로 버그</strong>: 참조 경로와 실제 파일 위치가 다름 — 수정 완료</li>
<li><strong>patient_id 매핑 오류</strong>: BASEINFO의 <code>id</code> 컬럼은 내부 순번(1~1407), <code>patient_number</code> 컬럼이 실제 차트 번호 — 이걸 혼동하면 환자 단위 GroupKFold가 잘못됨</li>
<li><strong>Multi_HL+HR 2,098 클립(16.1%)</strong>: category2_single에서 제외됐지만, 이 그룹은 pelvis meta가 Normal과 유사해 현재 모델로는 잡기 어려운 케이스</li>
</ul>

<p style="background:#eff6ff; padding:12px; border-radius:8px;"><strong>효과:</strong> 이후 새 실험 코드에서 데이터 접근을 <code>load_split()</code> 한 줄로 통일했습니다. 어떤 테스트셋을 썼는지, 어떤 라벨 스킴을 썼는지 코드에서 명시적으로 보입니다. 재현성이 구조적으로 보장됩니다.</p>
