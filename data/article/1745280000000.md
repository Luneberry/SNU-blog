<h1>관절 각도가 실제로 틀렸다 — aROM 검증과 데이터 인프라 구축</h1>

<p style="color:#6b7280; font-size:0.95em;">4월 중순은 새 실험을 추가하기 전에 기반을 점검한 시간이었습니다. aROM 계산에 물리적으로 불가능한 값(엉덩이 각도 266°)이 나오는 버그를 발견해 수정했고, 787개 클립을 대상으로 임상 문헌의 예측이 실제 데이터와 얼마나 맞는지 검증했습니다. 그리고 여러 프로젝트에 흩어져 있던 데이터를 하나의 API로 통합했습니다.</p>

<hr/>

<h2>배경: 아직 믿을 수 없는 숫자들</h2>

<p>모델에 관절 각도(aROM)를 추가하기 전에 먼저 확인해야 할 것이 있었습니다. 우리가 계산하는 각도가 실제로 올바른가? 수의학 문헌이 예측하는 파행 패턴이 실제 데이터에서도 나타나는가?</p>

<p>aROM은 "관절 운동 범위(Active Range of Motion)"입니다. 예를 들어 무릎이 보행 중 최소 얼마나 굽혀지고 최대 얼마나 펴지는지를 나타내는 값입니다. 파행이 있는 다리는 통증 때문에 무릎을 덜 굽히거나 더 굽히는 등 비정상적인 패턴을 보입니다.</p>

<h2>버그 발견: 엉덩이 관절이 266도?</h2>

<p>분석 중에 이상한 숫자를 발견했습니다. Normal(정상) 개의 엉덩이 관절 최대 각도가 266°로 계산됐습니다. 인간이든 개든 관절이 266° 꺾일 수는 없습니다.</p>

<p>원인을 추적해보니 각도 계산 코드의 모드 설정이 잘못되어 있었습니다. 각도 계산에는 두 가지 방식이 있습니다. "내각"을 그대로 쓰는 방식(default_mode)과 반대 방향 각도를 쓰는 방식(opposite_mode)입니다. 엉덩이와 발목 관절은 움직임 특성상 180° 초과가 가능하기 때문에 opposite_mode를 써야 하는데, default_mode가 적용되어 있었습니다.</p>

<h2>임상 시그니처 검증: 문헌이 맞았다</h2>

<p>버그를 수정한 후 787개 클립을 15개 그룹(정상, 왼쪽 앞다리 파행, 오른쪽 뒷다리 파행 등)으로 나눠 수의학 문헌이 예측하는 패턴이 실제로 나타나는지 검증했습니다.</p>

<p>수의학 문헌(Jaegger 2002, Millis &amp; Levine 2012)에 따르면 파행이 있는 다리의 관절은 통증으로 인해 비정상적인 각도를 보입니다. 예를 들어 앞다리 파행 시 팔꿈치가 덜 펴지고(굽힘 유지), 뒷다리 파행 시 무릎이 더 굽혀집니다.</p>

<p>대부분의 예측이 적중했지만, 고관절(엉덩이)은 변화가 없었습니다. 원인은 포즈 추정 모델의 한계 — 엉덩이 주변 관절(GREATER_TROCHANTER, ILIAC_CREST)의 위치 추정이 정확하지 않아 각도 신호가 노이즈에 묻힙니다.</p>

<h2>숨겨진 문제: 양측 파행이 "정상처럼" 보인다</h2>

<p>흥미로운 발견이 있었습니다. 양쪽 뒷다리 모두 파행이 있는 개(Multi_HL+HR, 전체 16.1%)는 골반 진폭이 Normal(정상)과 거의 같았습니다.</p>

<div style="background:#fef3c7; border-left:4px solid #f59e0b; padding:12px 16px; margin:16px 0;">
<strong>왜 양측 파행이 정상처럼 보이나?</strong><br/>
파행 탐지의 핵심 단서는 <em>좌우 비대칭</em>입니다. 왼쪽만 아프면 왼쪽과 오른쪽 움직임이 달라집니다. 그런데 양쪽이 다 아프면 좌우가 "대칭적으로" 나빠지기 때문에, 비대칭 지표(골반 진폭 등)가 정상처럼 보입니다. 현재 모델이 이 케이스를 제대로 잡으려면 <em>절대적인 관절 각도 패턴</em>을 봐야 합니다. 실제로 양측 파행에서는 양쪽 무릎이 동시에 4° 더 굽혀지는 패턴이 확인됐고, 이는 관절 각도 피처로만 포착 가능합니다.
</div>

<p>통계 분석(ANOVA)에서도 확인됐습니다. 상위 7개 피처는 모두 기존 meta(골반/머리 운동 지표)였고, 8~12위에 처음으로 관절 각도 피처들이 진입했습니다. 관절 피처는 기존 meta와 독립적인 정보를 담고 있어, 추가 시 모델에 새로운 정보가 들어갑니다.</p>

<h2>데이터 인프라: 모두가 같은 문을 통해 들어오게</h2>

<p>이 시기에 병행해서 진행한 작업이 데이터 통합입니다. Project 0, 1, 3, 4, 5가 각자 다른 경로에서 데이터를 읽고 있었습니다. 새 실험을 만들 때마다 어디서 어떤 파일을 읽어야 하는지 찾아야 했습니다.</p>

<p><code>dog/data/</code>를 정본으로 만들고 단일 진입점 API를 구축했습니다:</p>

<pre><code>from data.loader import load_split

# 단일파행 3-class (왼뒷/오른뒷/정상)
train, test = load_split('category2_single', version='v2')

# 신경계 vs 정형외과 2-class
train, test = load_split('category1_neu', fold=0)

# 관절 각도 포함 풍부한 dict 형태
train, test = load_split('category2_single', rich=True)</code></pre>

<p>기존 코드들은 심볼릭 링크로 호환성을 유지하므로 수정 없이 그대로 동작합니다. 이 인프라가 이후 모든 실험의 기반이 됩니다.</p>
